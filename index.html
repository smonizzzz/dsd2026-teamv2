01 · Context

Why This Project Exists

Recovering from limb surgery is rarely straightforward. Traditional rehabilitation demands frequent clinic visits and constant supervision — a model that is costly, time-consuming, and inaccessible for many patients. Our project tackles this gap head-on by building an intelligent, sensor-driven assistant that guides patients through their recovery at home.

Developed jointly by UTAD and Jilin University as part of the Distributed Software Development course, the system brings together 28 students across 6 specialised teams, each contributing a distinct technical layer to one unified rehabilitation platform.

07 · IF-S2-V2

Interface Conflict Log

During System Design integration, S2 (Zhiqi Zhang) identified 9 conflicts between the S2 Interface Specification and V2’s actual backend. All conflicts are within IF-S2-V2. Below is V2’s proposed resolution for each one.

接口冲突
Conflict 1 Endpoint Path Mismatch Type B Resolved
S2 Design
  • POST /api/session
  • POST /api/session/<sessionId>/data
  • PUT /api/session/<sessionId>/end
V2 Actual
  • POST /sessions
  • POST /measurements or /measurements/batch
  • PATCH /sessions/:id/end
V2 Resolution

S2 adopts V2’s path structure. No /api prefix is used. Measurements are a top-level resource, not nested under sessions — this cleanly separates concerns and allows other teams (V1, M1) to query measurements independently. S2 updates their implementation to call /sessions and /measurements directly.

Conflict 2 HTTP Method Mismatch — End Session Type B Resolved
S2 Design

Uses PUT to end a session.

V2 Actual

Uses PATCH — only one field (ended_at) is updated, not the full resource.

V2 Resolution

Use PATCH. This is semantically correct per HTTP spec: PUT replaces the entire resource, PATCH applies a partial update. S2 updates their client to send PATCH /sessions/:id/end.

Conflict 3 Session Creation Request Body Type A Resolved
S2 Sends
{ "sessionId": "session-001", "patientId": "patient-001", "sensorJointMapping": {...}, "payloadStatus": "bend_knee_10", "startTime": 1714456800000 }
V2 Expects
{ "userId": 1 }
V2 Resolution
  • V2 ignores unknown fields — S2 may send extra fields without breaking the API.
  • patientId → userId: S2 renames this field to userId and uses the integer ID returned by GET /users. String IDs are not supported.
  • sessionId: Server generates the ID. S2 must not provide it; use the id returned in the response for subsequent calls.
  • startTime: Server auto-generates started_at. Client-provided start times are ignored for now.
  • V2 returns 201 Created on success (not 200 OK).
Conflict 4 Data Upload Structure — Critical Type C In Negotiation
S2 Sends
{ "sensorData": [ { "sensorId":"...", "accX":..., "accY":..., "accZ":..., "gyroX":..., "gyroY":..., "gyroZ":..., "roll":..., "pitch":..., "yaw":... } ], "targetAngles": [ { "angleID":"...", "angle":... } ], "errors": [ { "sensorId":"...", "errorType":"...", "message":"..." } ] }
V2 Expects
{ "sessionId": 1, "jointAngles": { "knee": 45.2, "hip": 30.1 }, "isCorrect": false }
V2 Proposed Resolution

V2 will create a dedicated ingestion endpoint POST /ingest that accepts S2’s full payload. Internally, V2 will:

  • Map targetAngles → joint_angles (keyed by angleID)
  • Store raw IMU data in a new raw_imu JSON field on the measurements table
  • Store sensor errors in a separate sensor_errors JSON field
  • Set is_correct = false by default; V1 fills this in later via their own endpoint

⚠ This requires V1 input: V1 must confirm how they consume the data before V2 finalises the schema extension. Joint call between S2 + V1 + V2 required.

Conflict 5 Session End Request Body Type A Resolved
S2 Sends
{ "endTime": 1714456860000, "sampleCount": 1200, "errorCount": 3 }
V2 Actual

No body expected. Server auto-sets ended_at = now(). No fields for sampleCount or errorCount.

V2 Resolution

V2 will accept the optional body but ignore unknown fields for now. sampleCount and errorCount are computable server-side from existing data and do not need to be stored separately. endTime is ignored — server timestamp is authoritative to avoid clock skew between devices.

Conflict 6 Timestamp Format Type B Resolved
S2 Design

Unix milliseconds integer — e.g. 1714456800000

V2 Actual

ISO 8601 UTC string — e.g. "2025-04-21T14:30:00.000Z"

V2 Resolution

All timestamps use ISO 8601 UTC throughout the system. This is the REST API standard, human-readable, and timezone-unambiguous. S2 converts Unix ms to ISO 8601 before sending. V2 will accept both formats on the optional timestamp field and normalise internally if needed.

Conflict 7 Field Naming Convention Type B Resolved
S2 Design

camelCase in both requests and responses.

V2 Actual

camelCase in requests (what you send), snake_case in responses (what you receive back).

V2 Resolution

The mixed convention is intentional and already documented in V2’s integration guide. All teams must follow it: send camelCase, parse snake_case. This is the single most common cause of bugs during integration — read the guide carefully. V2 will not change response casing, as it would break teams already built against it.

Conflict 8 Session ID Type Type B Resolved
S2 Design

String, client-generated — e.g. "session-001"

V2 Actual

Integer, server auto-incremented — e.g. 1, 2, 3

V2 Resolution

Server-generated integer IDs are used throughout. Client-provided IDs are rejected. S2 must call POST /sessions, receive the id integer in the response, and use that value in all subsequent measurement uploads. This prevents ID collisions across distributed clients and is consistent with the rest of the API.

Conflict 9 HTTP Status Codes & Authentication Type B Resolved
S2 Design
  • Success: 200 OK for all operations
  • Auth failure: 401 Unauthorized
V2 Actual
  • Resource creation: 201 Created
  • Reads/updates: 200 OK
  • No authentication — 401 is never returned
V2 Resolution

V2’s status codes follow HTTP semantics: 201 for new resource creation, 200 for reads and updates. Authentication is not implemented in Sprint 1; 401 responses do not exist yet. When auth is added in Sprint 3, a 401 scheme will be agreed with all teams simultaneously. S2 should treat any unexpected 401 as a server misconfiguration for now.

待定

1 conflict remains open (Conflict 4) — the raw IMU data structure requires a joint call between S2, V1, and V2 before V2 can finalise the schema extension and the /ingest endpoint. All other 8 conflicts are resolved from V2’s side.

2. Scope

V2 — Backend API & Storage is the central server-side component of the Limb Motion Recognition and Assistant system. It is implemented as a Node.js + Express REST API backed by a SQLite database (sql.js). V2 is the single integration point for all other subsystems: it receives sensor measurement data from S2, serves session and measurement data to M1 and M2, provides AI classification inputs to V1, and manages all persistent storage for the platform.

V2 does not have a direct user-facing interface. It provides the following services to other groups:

  • User registration, authentication (JWT), and clinician approval workflow
  • Session lifecycle management (create, close)
  • Measurement ingestion (single and batch) and retrieval with date-range filtering
  • Patient list and profile access for the doctor dashboard
  • Rehabilitation recommendation management (create, read, update, delete)
  • Rehabilitation schedule storage and retrieval
  • Push notification token registration and dispatch

The following are out of scope for V2: sensor hardware and firmware (S1), raw sensor data acquisition and mock generation (S2), patient mobile application UI (M1), doctor web dashboard UI (M2), and AI motion classification model training (V1).

3. Glossary

TermDefinition
SessionA bounded rehabilitation exercise event with a started_at timestamp and an optional ended_at. Measurements may only be added to an open (not ended) session.
MeasurementA single joint-angle reading associated with a session. Contains joint_angles (JSON object), is_correct (boolean, set by V1), and an ISO 8601 timestamp.
PendingClinician account status after self-registration, before admin approval. A pending account cannot log in.
ActiveFully operational user account status. Required for login.
JWTJSON Web Token — stateless bearer token issued by V2 on successful login, required on all protected endpoints.
ISO 8601Timestamp format used throughout V2: YYYY-MM-DDTHH:MM:SS.sssZ. All date fields are normalised to this format on every database read.

3. Specific Use Cases

IUC-V2-01 Patient Registration Actor: Patient · M1
M1 uses the "Patient Registration" use case of V2. In this use case, a patient opens M1 and fills in their credentials. M1 sends a registration request containing username, password, email, role=patient, and an optional age field to V2's POST /auth/register endpoint as a JSON body. V2 hashes the password using bcryptjs, creates a new user record in the database with status=active, generates a signed JWT token, and returns it to M1 alongside the user profile. M1 stores the token and the patient gains immediate access to the application.
IUC-V2-02 Clinician Registration Actor: Doctor · M2
M2 uses the "Clinician Registration" use case of V2. In this use case, a doctor fills in their credentials and attaches a professional license file (PDF or image, maximum 5 MB) in M2's registration form. M2 sends a multipart/form-data request to V2's POST /auth/register endpoint. V2 stores the license file on disk under data/licenses/, creates a user record with status=pending, and returns a confirmation message to M2 without issuing a JWT token. The clinician must wait for a System Administrator to approve the account before being able to log in.
IUC-V2-03 User Login Actor: Patient · Doctor · M1 · M2
M1 and M2 use the "User Login" use case of V2. In this use case, the user submits their username and password through their respective application. The application sends these credentials to V2's POST /auth/login endpoint. V2 looks up the user record, verifies the password against the stored bcrypt hash, and checks that the account status is active. If both checks pass, V2 returns a signed JWT token encoding userId and role. If the account is pending, V2 rejects the request with a 403 response. The client stores the token and includes it in subsequent requests as a bearer token.
IUC-V2-04 Clinician Approval Actor: System Administrator · M2
M2 uses the "Clinician Approval" use case of V2. In this use case, a System Administrator reviews a pending clinician registration in M2's admin panel. The administrator selects the clinician and confirms the approval. M2 sends a PATCH /auth/approve/:userId request to V2 with the administrator's JWT token in the Authorization header. V2 verifies the token, locates the user record, sets status=active, and returns a confirmation to M2. The clinician can now log in successfully.
IUC-V2-05 Create Rehabilitation Session Actor: M1 · S2
M1 uses the "Create Rehabilitation Session" use case of V2. In this use case, when a patient begins an exercise in M1, M1 sends a POST /sessions request to V2 containing the patient's userId. V2 validates that the user exists, creates a new session record with a server-generated integer id and a started_at timestamp in ISO 8601 UTC, and returns the session object to M1. M1 stores the returned session id and provides it to S2. S2 uses this id to associate all subsequent measurement uploads with the correct session.
IUC-V2-06 End Rehabilitation Session Actor: M1 · S2
M1 uses the "End Rehabilitation Session" use case of V2. In this use case, when the patient finishes their exercise, M1 sends a PATCH /sessions/:id/end request to V2 with the session's integer id. V2 verifies the session exists and is still open, sets ended_at to the current ISO 8601 UTC timestamp, and returns the updated session object. From this point, V2 rejects any attempt to add new measurements to this session with a 409 Conflict response.
IUC-V2-07 Record Measurement(s) Actor: S2 · M1
S2 uses the "Record Measurement(s)" use case of V2. In this use case, after S2 processes raw IMU data from S1 into formal sensor data, S2 sends the data to V2. For individual measurements, S2 sends a POST /measurements request with sessionId and jointAngles (a JSON object keyed by joint name). For high-frequency streaming, S2 sends a POST /measurements/batch request with an array of measurement objects in a single atomic transaction. V2 validates that the target session is open, inserts the record(s) into the database, and returns confirmation to S2. The is_correct field defaults to false and is later updated by V1 after AI classification.
IUC-V2-08 Query Measurements Actor: M1 · M2 · V1
M2 uses the "Query Measurements" use case of V2. In this use case, a doctor opens a patient's session detail view in M2. M2 sends a GET /measurements/:sessionId request to V2, optionally including ?startDate= and ?endDate= query parameters to filter by date range. V2 looks up the session, applies the date filters if provided (accepting both YYYY-MM-DD and full ISO 8601 formats, normalised internally), and returns the filtered measurement array ordered by timestamp ascending to M2. M2 uses this data to render motion visualisation charts. V1 uses the same endpoint to retrieve measurements for AI classification.
IUC-V2-09 Access Patient Records Actor: Doctor · M2
M2 uses the "Access Patient Records" use case of V2. In this use case, a doctor opens the patient management panel in M2. M2 sends a GET /patients request to V2, which returns the list of all users with role=patient, including their id, username, email, age, and status. To view a specific patient's full profile, M2 sends a GET /patients/:id request. V2 returns the patient object, or a 404 response if the user does not exist or is not a patient. M2 displays this information in the doctor's dashboard.
IUC-V2-10 Manage Recommendations Actor: Doctor · M2 · Patient · M1
M2 uses the "Manage Recommendations" use case of V2. In this use case, after reviewing a patient's motion data, a doctor writes a rehabilitation recommendation in M2. M2 sends a POST /recommendations request to V2 with the patient's userId, the recommendation content, and an optional notes field for supplementary free-text. V2 stores the record and returns it with a server-generated integer id and the correct integer userId. The doctor may subsequently update the recommendation via PUT /recommendations/:id or delete it via DELETE /recommendations/:id. M1 retrieves the recommendations for a patient via GET /recommendations/:userId to display them to the patient. V2 returns an empty array (not a 404) if no recommendations exist yet.
IUC-V2-11 Manage Schedule Actor: Doctor · M2 · Patient · M1
M2 uses the "Manage Schedule" use case of V2. In this use case, a doctor assigns a rehabilitation schedule to a patient via M2. M2 sends a POST /schedule request to V2 with the patient's userId, a scheduledAt timestamp in ISO 8601 format, and a description of the planned exercise. V2 stores the entry and returns it to M2. M1 retrieves the patient's schedule via GET /schedule/:userId and displays upcoming sessions to the patient in the mobile application.
IUC-V2-12 Register Push Token Actor: M1
M1 uses the "Register Push Token" use case of V2. In this use case, after a patient successfully logs in, M1 obtains the device's push notification token from the operating system. M1 sends a POST /push/register request to V2 containing the userId and the device token. V2 stores the token associated with the user account and returns a confirmation including both user_id and token. This token is used in IUC-V2-13 to deliver push notifications to the patient's device.
IUC-V2-13 Send Push Notification Actor: M2 · V1
M2 uses the "Send Push Notification" use case of V2. In this use case, when a doctor publishes a new recommendation or when V1 generates a real-time feedback alert, the triggering system sends a POST /push/send request to V2 with the target userId and the notification payload. V2 looks up the device push token registered for that user via IUC-V2-12 and forwards the payload to the external push notification service. V2 returns a delivery confirmation or a 404 if no token has been registered for the user.

Part 4 — Others

1. Assumptions and Dependencies

  • Each patient has exactly one active account. The system does not handle duplicate accounts for the same person.
  • V2 does not validate the clinical content of recommendations. Clinical correctness is the responsibility of the doctor using M2.
  • The admin role responsible for approving clinician accounts (IUC-V2-04) is a real user in the system. Full role-based access control (RBAC) enforcement is planned for Sprint 3 and is not enforced at the route level in Sprint 2.
  • The system is an academic prototype. No GDPR or health data compliance obligation is assumed for this phase.
  • The server requires Node.js ≥ 18. The SQLite database is managed by sql.js (pure JavaScript — no native module compilation). The database file is located at data/v2.db and is created automatically on first startup.
  • All inter-team communication uses HTTP REST in Sprint 2. A WebSocket channel for real-time M1 streaming will be added in Sprint 3 without breaking existing REST endpoints.
  • All timestamps exchanged between teams use ISO 8601 UTC format. V2 normalises any SQLite-format dates automatically on every database read via a centralised helper function.
  • License files submitted during clinician registration are stored on the local filesystem at data/licenses/. Cloud storage integration is not in scope for Sprint 2.
  • The interface contract between V2 and S2 (IF1) is frozen for all conflicts except Conflict 4, which is pending a joint S2 + V1 + V2 decision on the raw IMU ingestion schema.
  • The external push notification service and its credentials are configured via environment variables. V2 does not select or manage the third-party push provider.

2. Non-Functional Requirements

Performance

  • Simple read endpoints (GET /patients, GET /measurements/:sessionId) must respond within 200 ms for datasets up to 10,000 records.
  • Batch measurement inserts (POST /measurements/batch) must complete within 500 ms for batches of up to 500 records.
  • The server must handle at least 10 concurrent connections during integration testing without degradation in response time.

Security

  • Passwords must never be stored in plaintext. All passwords are hashed using bcryptjs with a cost factor of 10 or higher before being written to the database.
  • All protected API endpoints require a valid JWT bearer token in the Authorization header. Expired or malformed tokens must result in a 401 Unauthorized response.
  • License file uploads are limited to 5 MB. The server-side path of the stored license file (license_path) must never be included in any API response to prevent filesystem path disclosure.

Reliability

  • The database schema is initialised automatically each time the server starts. ALTER TABLE migration statements are wrapped in try/catch blocks so the server starts correctly on both a fresh database and an existing one.
  • Batch measurement inserts are executed as a single atomic transaction. A failure on any individual record must roll back the entire batch.
  • The GET /health endpoint must return 200 OK at all times when the server process is running, with no authentication required.

Maintainability

  • The codebase follows a strict MVC pattern: routes, controllers, middleware, and database helpers are placed in separate directories. No business logic is permitted in route definition files.
  • Date normalisation is implemented exclusively in src/db/helpers.js (normalizeDates). No individual controller performs date conversion.
  • All commit messages and code comments are written in English.
  • No native module compilation is required. The project must be fully installable via a standard npm install on any platform.
