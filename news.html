Three API Bugs Patched After Cross-Team Integration Testing

Integration testing by the project coordination team identified three consistency issues in the V2 API. All three were reviewed against the full codebase and corrected in a single focused session.

"We read every controller, every route, every helper before touching anything. These fixes were surgical — one line each."

Bug 1 — userId type: GET /recommendations/engine/:userId was returning userId as a string instead of an integer, inconsistent with every other endpoint. Fixed with Number(). Bug 2 — snake_case: POST /push/register response used camelCase userId while the entire API uses snake_case user_id in responses. Fixed. Bug 3 — ISO 8601 dates: created_at, started_at, and ended_at were returning SQLite's native format ("2026-05-02 13:43:28") while other fields returned proper ISO 8601. A centralised normalizeDates function was added to db/helpers.js — it automatically converts every *_at field in every query to ISO 8601, covering both existing and future records.

Clinician Registration Now Requires Admin Approval — Patient Flow Untouched

M2's medical professionals operate under a new two-step identity verification workflow. Clinician accounts enter a pending state after registration and cannot log in until an admin activates them.

Clinicians register via POST /auth/register sending multipart/form-data (with an optional license file field). The server responds with {"userId": 1, "status": "pending"} — no JWT is issued. Attempting to log in while pending returns 403 Account pending approval. An admin calls PATCH /auth/approve/:userId to activate the account; from that point login works normally and the user object includes the status field. The new GET /auth/status endpoint lets any authenticated user check their current approval state. Patients are completely unaffected — they register via JSON body and receive a token immediately, as before.

S2 Delivers IF-S2-V2 Conflict Report — V2 Responds with Resolutions

Zhiqi Zhang (S2) submitted a formal conflict analysis identifying 9 interface mismatches between the S2 specification and V2’s actual backend. Team V2 has reviewed all conflicts and issued resolutions for 8 of them.

The conflicts span endpoint paths, HTTP methods, request body structure, data granularity, timestamp format, field naming, session ID type, and authentication. Most are classified as Type B (easy to unify) and have been resolved in favour of V2’s existing design, which is already in use and documented in the integration guide. One critical conflict remains open: Conflict 4 concerns the raw IMU data upload structure, where S2’s payload (accelerometer, gyroscope, orientation, errors) is fundamentally different from V2’s current measurements schema. This requires a joint call between S2, V1, and V2 before V2 can finalise a new /ingest endpoint and schema extension.

🇨🇳 Zhiqi Zhang · S2  |  🇵🇹 Sergio Moniz · V224 Apr 2026
Conflict Detail
✔ 8 Resolved

Eight of Nine Conflicts Closed: Paths, Methods, IDs, Timestamps, and Naming All Unified

V2 issued clear resolutions for the straightforward conflicts, establishing the conventions that S2 must follow when implementing their client.

"Server-generated integer IDs. ISO 8601 timestamps. PATCH not PUT. No /api prefix. These are non-negotiable — they are already live."

Conflicts 1, 2, 6, 7, 8, and 9 were resolved outright: S2 adopts V2’s path structure, uses PATCH for session end, sends ISO 8601 timestamps, reads snake_case responses while sending camelCase requests, uses server-assigned integer session IDs, and expects 201 on creation. Conflicts 3 and 5 (richer session creation body and session-end metadata) are resolved by V2 silently ignoring unknown fields — S2 may send extra data without breaking the API.

Conflict 4: Raw IMU Data Structure Requires Tri-Team Negotiation

S2’s sensor payload contains raw accelerometer, gyroscope, orientation, and error data that V2’s current schema cannot store. This is the sole remaining blocker.

V2 has proposed a /ingest endpoint that maps S2’s targetAngles to joint_angles and stores raw IMU data in an extended JSON field. V1 must confirm what fields they need before V2 finalises the schema. A joint call between all three teams is required.

System Design 1.0 Locks V2 as the Central Node.js/Express Backend

On 13 April, Team V2 closed the first system design revision, defining the front-end, intermediate, and back-end responsibilities for the distributed platform.

The revised design assigns the mobile app and IMU sensors to the front-end, places the V2 backend in the intermediate layer, and keeps SQLite persistence plus the V1 AI module in the back-end. V2 now owns the REST API surface, including /health, /users, /sessions, /measurements, and /recommendations, with controllers, routes, middleware, and database access arranged in an MVC-like structure. The result is a stable contract for sensor ingestion, AI feedback, and recommendation delivery.
